# ADR-006: Circuit Breaker для LLM Провайдеров

**Статус:** Принято

**Дата:** 2026-07-12

## Контекст

LLM-провайдеры (Ollama, OpenAI, GigaChat) — это внешние зависимости, которые могут отказывать или работать медленно. Отказ LLM не должен останавливать всё приложение. Системе нужно корректное ухудшение (graceful degradation).

## Решение

Принять паттерн **Circuit Breaker** для всех вызовов LLM-провайдеров:

### Состояния
| Состояние | Поведение |
|-----------|-----------|
| **CLOSED** | Нормальная работа — запросы проходят |
| **OPEN** | Все запросы быстро отклоняются (без вызова провайдера) |
| **HALF_OPEN** | После recovery timeout — разрешён один тестовый запрос |

### Конфигурация
| Параметр | По умолчанию | Описание |
|----------|-------------|----------|
| failure_threshold | 5 | Последовательных отказов для открытия цепи |
| recovery_timeout | 30s | Время до перехода в half-open |
| half_open_max_requests | 1 | Запросов разрешено в half-open состоянии |

### Fallback
- Circuit возвращает `fallback_response` при OPEN
- Coach Agent возвращает нулевые оценки при недоступности LLM
- Simulator Agent переключается на rule-based ответы

## Рассмотренные альтернативы

- **Только retry:** Не защищает от каскадных отказов — отклонено.
- **Без circuit breaker:** Отказ LLM блокирует все запросы — отклонено ради отказоустойчивости.
- **Только rate limiting:** Не защищает от длительных отказов провайдера — отклонено.

## Последствия

- **Положительно:** Приложение переживает отказы LLM-провайдеров
- **Положительно:** Быстрый отказ (без ожидания timeout) при OPEN состоянии
- **Положительно:** Автоматическое восстановление при возвращении провайдера
- **Отрицательно:** Дополнительная сложность — необходимо отслеживать состояние на провайдер
- **Отрицательно:** Возможны ложные срабатывания при перемежающихся отказах
