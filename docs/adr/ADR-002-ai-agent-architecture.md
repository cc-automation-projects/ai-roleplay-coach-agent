# ADR-002: Архитектура AI-агентов

**Статус:** Принято

**Дата:** 2026-07-12

## Контекст

Приложению нужны AI-агенты (Simulator, Coach, Curator, Analyst, Gamification), которые могут надёжно работать без внешних LLM-зависимостей. Разные пользователи могут хотеть разные LLM-бэкенды (Ollama, OpenAI, GigaChat).

## Решение

Принять **модульную архитектуру агентов**:

1. Каждый агент — независимый класс с единственной ответственностью
2. Агенты используют **rule-based логику** по умолчанию (LLM не требуется)
3. LLM-варианты используют **Provider Factory**:
   - `LLM_PROVIDER=mock` — детерминированные ответы, ноль зависимостей
   - `LLM_PROVIDER=ollama` — локальная LLM через Ollama
   - `LLM_PROVIDER=openai` — OpenAI API
   - `LLM_PROVIDER=gigachat` — GigaChat API
4. Будущее: миграция на **LangGraph** для stateful оркестрации агентов

## Рассмотренные альтернативы

- **Только LLM:** Требует от всех пользователей запуска LLM — отклонено из-за доступности.
- **Единый монолитный агент:** Сложнее тестировать и расширять — отклонено для поддерживаемости.
- **Микросервис на агента:** Слишком тяжело для текущего масштаба — отклонено из-за операционной сложности.

## Последствия

- **Положительно:** Работает без установки любой LLM
- **Положительно:** Легко добавить нового LLM-провайдера — один фабричный метод
- **Положительно:** Каждый агент может тестироваться независимо
- **Отрицательно:** Rule-based агенты менее гибки, чем LLM-агенты
- **Отрицательно:** Необходимо поддерживать оба кодовых пути (rule-based и LLM)
