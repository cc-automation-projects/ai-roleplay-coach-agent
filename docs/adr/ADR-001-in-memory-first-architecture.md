# ADR-001: In-Memory-First архитектура

**Статус:** Принято

**Дата:** 2026-07-12

## Контекст

Проекту нужен режим разработки с нулевой конфигурацией. Новые разработчики должны иметь возможность клонировать репозиторий и запустить приложение без настройки PostgreSQL, Redis, Qdrant или любых внешних сервисов. В то же время архитектура должна поддерживать полноценную PostgreSQL-персистентность для production-развёртываний.

## Решение

Принять **in-memory-first** архитектуру:

1. По умолчанию `DB_MODE=memory` — все данные хранятся в Python dict
2. Опционально `DB_MODE=postgres` — полная PostgreSQL-персистентность
3. Repository Pattern абстрагирует доступ к данным — единый код для обоих режимов
4. Seed data (3 пользователя, 3 сценария) загружаются при запуске в обоих режимах

## Рассмотренные альтернативы

- **Только PostgreSQL:** Заставил бы каждого разработчика запускать PostgreSQL — отклонено из-за трения при онбординге.
- **SQLite:** Легче PostgreSQL, но всё равно требует настройки файловой системы и не тестирует тот же SQL-диалект, что в production.
- **Mock всего:** Слишком далёк от реального поведения — in-memory репозитории исполняют реальную логику через тот же интерфейс.

## Последствия

- **Положительно:** Ноль внешних зависимостей для dev — clone, `pip install -e .`, `python main.py`
- **Положительно:** Один и тот же код для memory и postgres режимов — нет расхождения веток
- **Положительно:** Тесты по умолчанию работают в memory-режиме — быстрее и надёжнее
- **Отрицательно:** Необходимо поддерживать две реализации репозитория на каждую сущность
- **Отрицательно:** In-memory режим не тестирует PostgreSQL-специфичные возможности (например, полнотекстовый поиск, оконные функции)
