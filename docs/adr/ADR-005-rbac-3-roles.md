# ADR-005: RBAC с 3 ролями

**Статус:** Принято

**Дата:** 2026-07-12

## Контекст

Разные пользователи нуждаются в разных уровнях доступа. Операторам нужна практика сессий, тренерам — управление группами и назначениями, администраторам — доступ к конфигурации системы.

## Решение

Принять **3-ролевую RBAC** модель:

| Роль | Разрешения |
|------|-----------|
| **operator** | CRUD своих сессий, просмотр своих оценок, свои данные геймификации, свой профиль |
| **trainer** | Все разрешения operator + просмотр данных группы операторов, создание/назначение учебных планов, групповая аналитика |
| **admin** | Все разрешения trainer + управление пользователями, управление сценариями, просмотр всех данных, конфигурация системы |

### Реализация
- JWT токены (`access_token` 30 мин, `refresh_token` 7 дней)
- Декоратор `require_role()` на endpoint'ах
- Иерархия ролей в middleware: `admin >= trainer >= operator`
- Token blacklist при logout (Redis)

## Рассмотренные альтернативы

- **Плоские разрешения (без ролей):** Гибко, но неуправляемо в масштабе — отклонено.
- **Тонкозернистые разрешения (per-endpoint):** Слишком сложно для 3-ролевой системы — отклонено.
- **OAuth2 с внешним провайдером:** Over-engineering для текущего масштаба — отклонено.

## Последствия

- **Положительно:** Чётко, просто, легко аудитировать
- **Положительно:** Иерархия ролей сокращает избыточные проверки
- **Положительно:** JWT stateless — не требуется обращение к БД на каждый запрос
- **Отрицательно:** Добавление новой роли требует изменения кода и миграции
- **Отрицательно:** Обновление токенов требует клиентской обработки
